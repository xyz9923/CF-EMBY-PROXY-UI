# 全局设置功能文档

> 适用对象：第一次接触这个项目、对 Cloudflare / Worker / 代理参数不熟的用户  
> 对应页面：`/admin` -> `全局设置`

---

# 1. 先用一句话讲清楚：什么是“全局设置”？

“全局设置”就是这套 Worker 的**总控制面板**。

你在这里改的内容，不是只影响某一个节点，而是会影响：

- 整个管理后台怎么刷新
- 整个代理链路怎么转发
- 安全策略怎么拦截
- 日志怎么保存
- Telegram 怎么发通知
- Cloudflare 统计和清缓存怎么联动

可以把它理解为：

- **节点管理**：是“每台源站自己的设置”
- **全局设置**：是“整个系统共同遵守的设置”

---

# 2. 小白先记住这 6 条

如果你不想把系统改坏，先记住下面几点：

1. **不懂的先别改，默认值已经能用。**
2. **一次只改一个分区。**
3. **改完先保存，再实际播放测试。**
4. **涉及网络协议、超时、重试的选项，尽量小步调整。**
5. **改之前最好先导出“全局设置”或“完整备份”。**
6. **看到“恢复推荐值”时，不确定就先用它。**

---

# 3. 全局设置分为哪几块？

后台里一共分成 5 个分区：

1. `系统 UI`
2. `代理与网络`
3. `缓存与安全`
4. `日志与监控`
5. `账号与备份`

下面我会按这 5 块分别解释。

---

# 4. 系统 UI

这一栏目前主要控制的是：**仪表盘自动刷新**。

## 4.1 开启 Dashboard 自动刷新

**它是什么**

打开后，后台仪表盘会自动重新拉取：

- 统计面板
- 运行状态

**适合谁**

- 你经常开着后台看运行状态
- 你想把后台当“监控面板”

**不适合谁**

- 你只是偶尔登录看一眼
- 你担心后台请求太频繁

**小白建议**

- 不值班：可以关掉
- 需要观察：可以打开

## 4.2 自动刷新周期（秒）

**它是什么**

控制后台多久自动刷新一次。

**通俗理解**

- 数字越小，刷新越勤
- 数字越大，刷新越慢

**建议值**

- 小白推荐：`30` 到 `60`

**风险**

- 太小：后台请求会明显变多
- 太大：数据更新不及时

# 5. 代理与网络

这是最核心、也最容易让小白紧张的一栏。  
如果你不知道自己在做什么，**优先保持默认值**。

## 5.1 HTTP/2

**它是什么**

控制是否允许 Worker 与上游之间尝试使用 `HTTP/2`。

**什么时候可能需要开**

- 你明确知道自己的源站支持得很好
- 你正在针对某些网络环境做实验

**什么时候不要乱开**

- 你只是想“先稳定能看”
- 你并不清楚协议差异

**小白建议**

- 默认不要主动开启

## 5.2 HTTP/3 QUIC

**它是什么**

控制是否允许使用 `HTTP/3 / QUIC`。

**可能的好处**

- 某些网络环境下更快

**可能的问题**

- 弱网、复杂运营商线路下，稳定性不一定更好

**小白建议**

- 网络环境好可以试
- 出问题就先关掉，回到默认稳态

## 5.3 晚高峰自动降级为 HTTP/1.1

**它是什么**

晚上高峰时段，系统自动优先选择更稳的传输方式。

**为什么有这个功能**

晚高峰经常是最容易出现：

- 回源抖动
- 协议不稳定
- 视频播放异常

**小白建议**

- 保持开启

## 5.4 协议回退与 403 重试

**它是什么**

当上游返回 `403` 或协议握手不稳定时，系统会尝试：

- 去掉可疑头
- 换更稳的方式再试一次

**它解决什么问题**

- 某些 Emby / 源站在特殊请求头或协议组合下容易报错

**小白建议**

- 保持开启

## 5.5 开启视频起播预热拦截

**它是什么**

系统会识别播放器刚开始播放时的探测请求，提前做一点缓存和准备动作。

**你能理解成**

“让视频刚点开时更容易秒开，减少第一下卡顿。”

**小白建议**

- 保持开启

## 5.6 预热微缓存时长（秒）

**它是什么**

预热命中后，边缘缓存保留多久。

**影响**

- 太小：预热收益不明显
- 太大：没有必要地占用缓存

**小白建议**

- 保持默认 `180`

## 5.7 预热旁路预取字节数

**它是什么**

命中起播探测后，后台额外多读一点后续数据，为正式播放做准备。

**怎么理解**

- 数值越大，预热更激进
- 数值越小，预热更保守
- `0` 表示不做这部分额外预取

**小白建议**

- 保持默认
- 除非你明确遇到“起播慢”，否则不要乱调

## 5.8 源站和同源跳转代理

**它是什么**

控制源站 2xx 响应和同源 30x 跳转，是否继续由 Worker 中继。

**通俗理解**

- 开启：Worker 继续帮你转发
- 关闭：有些跳转会直接发给客户端

**小白建议**

- 保持开启

## 5.9 强制反代外部链接

**它是什么**

当出现外部跳转链接时，决定是否继续由 Worker 代理。

**通俗理解**

- 开启：尽量继续走你的 Worker
- 关闭：外部地址直接下发给用户

**小白建议**

- 保持开启

## 5.10 `wangpandirect` 关键词

**它是什么**

这是一个“直连关键词列表”。  
命中这些关键词的外部链接，会优先走直连，而不是继续由 Worker 中继。

**典型用途**

- 常见网盘
- 对象存储
- 某些本来就更适合直连的下载地址

**小白建议**

- 保持默认内置内容
- 只有你非常确定要加某类链接时再补关键词

## 5.11 源站直连名单

**它是什么**

你可以从节点列表里勾选某些节点，让它们在特定场景下直接回源，不再由 Worker 中继数据流。

**适合谁**

- 你已经很清楚哪些节点适合直连

**不适合谁**

- 你刚开始部署
- 你还不清楚“直连”和“反代”的差异

**小白建议**

- 先别勾
- 先全部保持 Worker 中继，更容易统一排障

## 5.12 Ping 超时时间

**它是什么**

后台探测节点健康时，等多久算超时。

**太小会怎样**

- 明明节点还活着，却被误判成慢或异常

**太大会怎样**

- 探测等待太久，后台反应慢

**小白建议**

- 保持默认 `5000`

## 5.13 上游握手超时

**它是什么**

Worker 等上游建立连接最多等多久。

**特殊值**

- `0` = 不启用这项保护

**小白建议**

- 不懂就保持默认
- 如果你怀疑上游“卡死不返回”，再考虑设置，比如 `30000`

## 5.14 额外重试轮次

**它是什么**

针对安全请求（比如 `GET` / `HEAD`），失败后额外再试几轮。

**通俗理解**

- `0` = 不额外重试
- `1` = 失败后再多试一轮

**小白建议**

- 先用默认值
- 网络偶发不稳定时，再试着设成 `1`

## 5.15 “恢复推荐值”按钮

**它是什么**

一键把这一栏常用参数恢复到一组比较稳妥的推荐值。

**适合什么时候用**

- 你改乱了，不知道怎么回去
- 你想先要一份“比较稳”的配置

**小白建议**

- 不确定时优先使用

# 6. 缓存与安全

这一栏主要决定：**谁能访问、访问频率、浏览器跨域、图片缓存时间**。

## 6.1 国家/地区白名单

**它是什么**

只允许指定国家/地区的用户访问。

**例子**

- 填 `CN,HK`：只允许中国大陆和香港访问

**什么时候用**

- 你只想让特定地区访问

**小白建议**

- 不确定就留空

## 6.2 国家/地区黑名单

**它是什么**

禁止指定国家/地区访问。

**例子**

- 填 `US`：屏蔽美国访问

**小白建议**

- 没明确需求就先别配

## 6.3 IP 黑名单

**它是什么**

直接屏蔽指定公网 IP。

**适合场景**

- 恶意刷接口
- 攻击源
- 明确要封禁的异常来源

**小白建议**

- 没抓到明确坏 IP 时，不用填

## 6.4 全局单 IP 限速

**它是什么**

限制同一个 IP 每分钟最多发多少请求。

**为什么有用**

- 防止暴力刷接口
- 防止异常流量爆发

**特殊值**

- `0` = 不限制

**小白建议**

- 不清楚需求就留空或 `0`
- 如果担心被刷，可以从 `600` 这种相对保守的值开始

## 6.5 图片海报缓存时长（天）

**它是什么**

只影响海报、封面等图片类资源的缓存时长。

**它不会影响什么**

- 不会直接改变视频主播放链路

**小白建议**

- 保持默认 `30`

## 6.6 CORS 跨域白名单

**它是什么**

限制哪些网页前端能在浏览器里跨域调用你的 Worker API。

**小白怎么理解**

- 如果你完全不知道 CORS 是什么，先别动
- 留空时通常更宽松

**适合什么时候配置**

- 你明确只允许某个网页域名访问

**例子**

- `https://emby.example.com`

# 7. 日志与监控

这一栏决定：**日志怎么写、多久保留、失败怎么重试、Telegram 怎么通知你**。

## 7.1 日志保存天数

**它是什么**

日志最多保留多少天，超过后系统自动清理。

**小白建议**

- 常见家庭使用：`7` 到 `14`

## 7.2 日志写入延迟（分钟）

**它是什么**

请求日志先放在内存队列里，过一段时间再批量写入 D1。

**怎么理解**

- 小一点：日志更快落盘
- 大一点：更省写入次数

**小白建议**

- 保持默认 `20`

## 7.3 日志提前写入条数阈值

**它是什么**

如果日志积累到一定数量，即使还没到延迟时间，也会提前写入。

**小白建议**

- 保持默认 `50`

## 7.4 D1 单批写入切片大小

**它是什么**

每次写数据库时，一批拆成多大的块。

**小白建议**

- 保持默认
- 只有在数据库写入卡顿时，再考虑调小

## 7.5 D1 失败重试次数

**它是什么**

写日志失败后，再补试几次。

**小白建议**

- 保持默认 `2`

## 7.6 D1 重试退避（毫秒）

**它是什么**

每次重试之间等多久。

**小白建议**

- 保持默认 `75`

## 7.7 定时任务租约时长

**它是什么**

用于避免多个定时任务同时重复执行。

**小白怎么理解**

- 可以理解为“这个定时任务先占个锁，别人先别重复跑”

**小白建议**

- 保持默认 `300000`

## 7.8 推荐生产值 / 异常调优指引

这一块其实就是给小白准备的“参考答案”。

你可以把它理解成：

- 绿色块：比较稳妥的参考区间
- 黄色块：出了问题先看哪里

如果你完全没经验：

- 先用默认值
- 不行再点“恢复推荐值”

## 7.9 Telegram Bot Token / Chat ID

**它是什么**

这是 Telegram 机器人通知需要的两项参数。

**有了它能做什么**

- 发测试通知
- 发每日报表
- 发异常告警

**没有会怎样**

- 不影响基本代理播放
- 只是通知类功能不能用

## 7.10 日志丢弃批次阈值

**它是什么**

当系统发现日志队列溢出、丢掉日志达到一定程度时，才发 Telegram 告警。

**特殊值**

- `0` = 关闭这个告警

## 7.11 D1 写入重试阈值

**它是什么**

日志写入数据库重试次数达到某个程度时，再触发告警。

**特殊值**

- `0` = 关闭这个告警

## 7.12 定时任务失败告警

**它是什么**

定时任务进入 `failed` 或 `partial_failure` 时发通知。

**小白建议**

- 如果你已经配好了 Telegram，可以开
- 没配 Telegram 就不要开

## 7.13 同类告警冷却时间

**它是什么**

相同类型的告警，多久内不要重复轰炸你。

**小白建议**

- 保持默认 `30`

## 7.14 发送测试通知

**它是什么**

立即发一条 Telegram 测试消息。

**什么时候用**

- 你刚填完 Bot Token / Chat ID
- 你想确认消息是不是能收到

## 7.15 手动发送日报

**它是什么**

立即触发一次日报发送。

**什么时候用**

- 你想测试日报格式
- 你刚配好 Telegram，想确认链路通不通

# 8. 账号与备份

这一栏主要是：**登录有效期、Cloudflare 联动、导入导出、快照恢复**。

## 8.1 免密登录有效天数（管理员 JWT）

**它是什么**

后台登录一次后，多久内可以保持登录状态。

**数字越大**

- 登录更省事
- 但长期不退出时，安全性会略弱一点

**小白建议**

- 保持默认 `30`

## 8.2 Cloudflare 账号 ID / Zone ID / API 令牌

**它们分别做什么**

- `账号 ID`：定位你的 Cloudflare 账号
- `Zone ID`：定位你的站点区域
- `API 令牌`：让 Worker 有权限读取统计、清理缓存等

**这些参数主要用于**

- 仪表盘增强统计
- 一键清理缓存
- 部分 Cloudflare 数据联动

**如果不填会怎样**

- 基础代理照样能跑
- 只是部分统计和联动功能不完整

**小白建议**

- 先把基础代理跑通
- 再补 Cloudflare 联动参数

## 8.3 一键清理全站缓存（Purge）

**它是什么**

清掉 Cloudflare 侧缓存。

**什么时候用**

- 你刚改了图片、静态页面、某些缓存策略
- 你怀疑旧缓存没刷新

**不要频繁乱点**

- 它不是“越点越好”
- 只是排障工具

## 8.4 导出完整备份 / 导入完整备份

**它是什么**

把“节点 + 全局设置”一起导出或导入。

**适合什么时候用**

- 换环境迁移
- 大改之前先留备份
- 想把整个系统搬到另一套 Worker

**小白建议**

- 每次大改前先导出完整备份

## 8.5 导出全局设置 / 导入全局设置

**它是什么**

只导入导出 settings，不带节点清单。

**适合什么时候用**

- 你有多个环境
- 你只想同步策略，不想覆盖节点列表

## 8.6 设置变更快照

**它是什么**

系统会自动保留最近的全局设置历史版本。

**有什么用**

- 改坏了可以回退
- 导入失败后也更容易恢复

**恢复快照时会发生什么**

- 当前配置会被替换
- 系统会先把“当前配置”再记一份快照
- 所以一般仍然能继续回退

**小白建议**

- 这是救命功能
- 改大配置前，先确认快照能看到

# 9. 小白最常见的 4 种用法

## 用法 1：我只想稳定播放，不想折腾

建议：

- 大部分保持默认
- `HTTP/2` 不开
- `HTTP/3` 不乱试
- `协议回退` 保持开启
- `预热` 保持开启
- `日志` 保持默认

## 用法 2：我只想看后台更像监控面板

建议：

- 打开 `Dashboard 自动刷新`
- 周期设成 `30` 或 `60`

## 用法 3：我想收到 Telegram 通知

步骤：

1. 填 `Bot Token`
2. 填 `Chat ID`
3. 点 `发送测试通知`
4. 能收到后，再考虑开启异常告警

## 用法 4：我怕改坏，想先保命

步骤：

1. 先 `导出全局设置` 或 `导出完整备份`
2. 再修改参数
3. 如果出问题，用 `设置变更快照` 恢复

# 10. 哪些选项最不建议小白乱动？

下面这些最容易改出“看起来懂了，其实把链路搞乱”的问题：

- `HTTP/2`
- `HTTP/3`
- `上游握手超时`
- `额外重试轮次`
- `源站直连名单`
- `wangpandirect` 自定义关键词
- `国家/地区白名单`
- `单 IP 限速`

如果你必须改，建议：

1. 先导出备份
2. 一次只改一个
3. 改完立刻播放测试

---

# 11. 哪些按钮最适合小白用？

最友好的几个按钮是：

- `保存`
- `恢复推荐值`
- `发送测试通知`
- `导出全局设置`
- `导出完整备份`
- `恢复此快照`

这些按钮的共同特点是：

- 成本低
- 容易验证结果
- 出问题也容易回退

---

# 12. 最后给小白的建议

如果你现在刚接触这个项目，最稳的顺序是：

1. 先别碰大多数高级网络参数
2. 先把基础播放跑通
3. 再配置 Cloudflare 联动
4. 再配置 Telegram 通知
5. 最后才去调超时、重试、直连名单

一句话总结：

**先求稳，再求快；先能用，再优化。**
